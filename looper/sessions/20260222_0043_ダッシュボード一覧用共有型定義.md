# 20260222_0043 ダッシュボード一覧用共有型定義

## タスク ID
dashboard-contracts

## 内容
ダッシュボード一覧用の共有型定義を実装。W2 以降の UseCase・UI タスクが参照する型を定義。

## 作成・変更したファイル

### 変更
- `apps/content-reviewer/src/backend/contexts/content-check/domain/gateways/content-check.repository.ts`
  - `ContentCheckFilter` に `source?: 'web' | 'slack'` を追加
  - `ContentCheckFilter` に `createdAfter?: Date` を追加
  - `ContentCheckFilter` に `createdBefore?: Date` を追加

### 新規作成
- `apps/content-reviewer/src/backend/contexts/content-check/application/usecases/content-check-list.dto.ts`
  - `ContentCheckListItemDto`: id, source, status, createdAt, segmentCount, summary を持つ plain object
  - `ContentCheckListDto`: items 配列を持つ DTO

## 設計判断

- DTOは architecture.md の規約に従い `application/usecases/` に配置
- ファイル名は既存の `content-check-detail.dto.ts` と同じ命名規約（kebab-case + `.dto.ts`）
- `source` フィールドは domain モデルの ContentCheckProps には現時点では存在しないが、filter の型定義としては追加済み（W2 タスクで domain モデルへの追加が必要になる可能性あり）
- `summary` は `{ error, warning, info }` のインライン型（ContentCheckSummaryDto の再利用を避け、detail.dto.ts との重複を許容）

## 詰まった点
特になし。単純な型定義追加。

## 次のタスクへの申し送り

- W2 の UseCase 実装で `ContentCheckListDto` を参照すること
- `ContentCheckFilter.source` を実際に利用するには `ContentCheck` domain model に `source` フィールドが必要になる可能性あり（現時点の model には `source` がない）
- `content-check-list.dto.ts` の `summary` 型は `content-check-detail.dto.ts` の `ContentCheckSummaryDto` と同一構造だが、意図的に分離している（DTO の独立性を保つため）
