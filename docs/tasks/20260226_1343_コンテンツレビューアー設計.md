# Content Reviewer 設計ドキュメント

## 目的

広報チームが、増加するSNS発信コンテンツの全件目視レビューから解放され、AIによる自動チェックで品質を担保できるようになるため。

### 背景

- 日本の政党が利用する内部ツール
- 議員増加に伴い発信（YouTube, X, note等）の分量が飛躍的に増加
- 注目度の上昇により、不適切な表現をなるべく避けたい状態
- 現在は広報チームが全件目視レビューしており、負荷が限界

### 解決策

LLMを活用したコンテンツ自動チェックシステムを構築し、以下の観点で文章を自動検証する:
- 一般的な事実のチェック（Web検索による）
- ナレッジコンテンツとの整合性確認（RAG検索）
- 表現ルール適合（登録済みルールとの照合）
- 炎上リスク判定（一般常識に照らした判定）
- 文章クオリティチェック（誤字脱字、繰り返し表現等）

- E2E テストは `.env.local` の実 API キーを使用する（実 AI Gateway + 実 DB）

## Bounded Context 設計

### 概要

```
contexts/
├── shared/              # 共通基盤（汎用型、AI/Embedding Gateway インターフェース）
├── auth/                # 認証（Google OAuth、セッション管理）
├── content-check/       # コンテンツチェック（分割・並列チェック・結果管理）
└── source-management/   # 参照データ管理（表現ルール・ナレッジ・ソース接続）
```

### 依存関係

```
content-check ──→ shared（AI Gateway, Embedding Gateway）
content-check ──→ source-management テーブル直接参照（infrastructure 層で expression_rules / knowledge_items テーブルを読む）
source-management ──→ shared（Embedding Gateway）
auth ──→ shared
```

> Context 間の import は composition → composition のみ許可。content-check の infrastructure 層が source-management の DB テーブルを直接読むのは、同一 DB 内のリードオンリークエリなので許容する（Gateway インターフェースは content-check の domain に定義）。


### auth Context

Google OAuth 認証とセッション管理。Supabase Auth のサーバーサイド PKCE フローを利用。

```
auth/
├── presentation/
│   ├── composition/
│   │   └── auth.composition.ts          # Supabase サーバークライアント生成
│   ├── loaders/
│   │   └── session.loader.ts            # セッション取得（Server Component 用）
│   └── actions/
│       └── logout.action.ts             # ログアウト処理
└── (domain/application/infrastructure は Supabase Auth に委譲するため最小限)
```

**認証フロー:**

1. `/login` ページ → 「Google でログイン」ボタン
2. ボタンクリック → `/api/auth/login` Route Handler
3. Route Handler → Supabase `signInWithOAuth` で Google OAuth URL 生成 → リダイレクト
4. Google 認証完了 → `/api/auth/callback` にコード付きリダイレクト
5. Callback → `exchangeCodeForSession` でセッション取得 → Cookie 設定 → `/` にリダイレクト
6. `(auth)/layout.tsx` → `session.loader.ts` でセッション検証、無効なら `/login` リダイレクト

**ドメイン制限:**
- Supabase プロジェクト設定で `team-mir.ai` ドメインに制限
- Callback でもメールドメインを二重チェック
- ドメインエラー時は `/login?error=domain_restricted` にリダイレクト

**開発環境スキップ:**
- `SKIP_AUTH=true` 時、ログイン画面に「開発用ログイン」ボタン表示
- `/api/auth/dev-login` → ダミーセッション（固定ユーザー）を Cookie に設定
- `(auth)/layout.tsx` の認証チェックも `SKIP_AUTH` 時はスキップ

**users テーブル連携:**
- 初回ログイン時に Supabase Auth の `auth.users` から `public.users` にレコード作成
- Callback Route Handler 内で upsert 実行

**AI チェック処理パイプライン:**

1. **テキスト分割**（`split-content.usecase.ts`）
   - AI に最大30,000字のテキストを渡し、レビューに適した論理セクションに分割
   - `generateObject` で `{ sections: [{ content: string }] }` を返す
   - セクションごとに `check_sections` レコードを作成（status: pending）

2. **並列チェック実行**（`run-check.usecase.ts`）
   - 各セクションに対して5つの UseCase を `Promise.all` で並列実行
   - 各チェック UseCase はセクションテキストを受け取り、指摘事項のリストを返す
   - 結果を `check_issues` に保存、セクション status を completed に更新

3. **各チェック UseCase の処理:**

   a. **事実確認**（`fact-check.usecase.ts`）
      - `AIGateway.generateObjectWithWebSearch` でテキスト内の事実主張を検証
      - Web 検索結果を根拠に、事実と異なる記述を指摘

   b. **ナレッジ整合性**（`knowledge-check.usecase.ts`）
      - `EmbeddingGateway.generateEmbedding` でセクションの embedding 生成
      - `KnowledgeSearchGateway.searchSimilar(embedding, limit: 5)` でナレッジ検索
      - 取得したナレッジとセクションを AI に渡し、不整合を検出

   c. **表現ルール**（`expression-check.usecase.ts`）
      - `ExpressionRuleQueryGateway.findAllEnabled()` で有効ルール全件取得
      - ルール一覧とセクションを AI に渡し、違反箇所を検出
      - issue に `ruleId` を含めて返す

   d. **炎上リスク**（`risk-check.usecase.ts`）
      - 日本の社会文化的文脈を踏まえたリスク分析
      - `AIGateway.generateObject` で判定

   e. **文章品質**（`quality-check.usecase.ts`）
      - 同一動詞の繰り返し、誤字脱字、文法エラーをチェック
      - `AIGateway.generateObject` で判定

4. **構造化出力スキーマ**（全チェック共通）:
   ```
   {
     issues: [{
       quote: string,        // 該当箇所の引用
       message: string,      // 指摘メッセージ
       suggestion?: string,  // 修正案
       severity: 'caution' | 'needs_fix',
       ruleId?: string       // 表現ルールチェックのみ
     }]
   }
   ```

**Slack 連携**（`/api/slack/events/route.ts`）:
- Slack Events API の `app_mention` イベントを受信
- URL verification challenge に対応
- `SLACK_SIGNING_SECRET` でリクエスト検証
- メンションされたスレッドのメッセージを取得
- `RunCheckUseCase` を実行（source: 'slack'）
- 完了後、スレッドにチェック結果サマリーとパーマリンクを投稿
- Vercel の `waitUntil` で非同期処理（Slack への即時 200 応答を維持）

**ExpressionRule モデル:**

- `enabled: boolean`（有効/無効）
- `createdAt: Timestamp`
- `updatedAt: Timestamp`

**KnowledgeItem モデル:**
- `id: string`
- `title: string`
- `sourceType: SourceType`（note / manifesto / manual）
- `sourceUrl?: string`
- `content: string`
- `embedding?: number[]`（1536次元）
- `sourceArticleId?: string`
- `createdAt: Timestamp`
- `updatedAt: Timestamp`

**embedding 生成フロー:**
- `create-knowledge.usecase.ts` 内で `EmbeddingGateway.generateEmbedding(content)` を呼び出し
- 生成された embedding を KnowledgeItem とともに保存
- 更新時も content 変更があれば embedding を再生成

**note.com 非公式 API 利用:**

- 記事一覧取得: `GET https://note.com/api/v2/creators/{username}/contents?kind=note&page={page}`
- 記事詳細取得: `GET https://note.com/api/v3/notes/{noteKey}`
- `ArticleFetchGateway` インターフェース:
  ```
  fetchArticleList(accountUrl: string): Promise<ArticleListItem[]>
  fetchArticleContent(articleUrl: string): Promise<string>
  ```

**記事取込フロー:**

1. ユーザーが「記事一覧を再取得」→ `sync-articles.usecase.ts`
   - note API から記事一覧を取得
   - 既存の `source_articles` と突合し、新規記事を追加
2. ユーザーがチェックボックスで記事を選択し「取り込む」→ `import-articles.usecase.ts`
   - 選択された記事の本文を note API から取得
   - 同一 Context 内の `CreateKnowledgeUseCase` でナレッジ作成 + embedding 生成
   - `source_articles.imported = true` に更新

### pgvector 設定

- Supabase で `vector` extension を有効化: `CREATE EXTENSION IF NOT EXISTS vector;`
- `knowledge_items.embedding` に `vector(1536)` 型を使用
- cosine similarity 検索用のインデックス: `CREATE INDEX ON knowledge_items USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);`
- Prisma では `Unsupported("vector(1536)")` 型で定義し、vector 検索は `$queryRaw` で実行

## 認証設計詳細

### Supabase Auth サーバーサイド PKCE

`@supabase/ssr` パッケージを使用し、サーバーサイドで認証処理を完結させる。

**サーバークライアント生成**（`auth.composition.ts`）:
- `createServerClient` で Cookie ベースのセッション管理
- Next.js の `cookies()` API を利用

**API Routes:**

1. `/api/auth/login/route.ts`:
   - `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo, queryParams: { hd: 'team-mir.ai' } } })`
   - 生成された OAuth URL にリダイレクト

2. `/api/auth/callback/route.ts`:
   - `supabase.auth.exchangeCodeForSession(code)` でセッション確立
   - ユーザーメールのドメイン検証（`team-mir.ai`）
   - `users` テーブルに upsert
   - ドメインエラー時: セッション削除 → `/login?error=domain_restricted`
   - 成功時: `/` にリダイレクト

3. `/api/auth/dev-login/route.ts`:
   - `SKIP_AUTH !== 'true'` なら 403
   - ダミーユーザー情報を Cookie に設定

**Middleware**（`middleware.ts`）:
- Supabase セッションの自動リフレッシュ
- `(auth)` 配下へのリクエストでセッション検証

## Slack 連携設計

### Slack App 設定

- Bot Token Scopes: `app_mentions:read`, `chat:write`, `channels:history`, `groups:history`
- Event Subscriptions: `app_mention` イベント
- Request URL: `https://{production-domain}/api/slack/events`

### 処理フロー

1. `app_mention` イベント受信
2. `SLACK_SIGNING_SECRET` でリクエスト署名を検証
3. URL verification challenge への応答（初回設定時）
4. スレッドの全メッセージを `conversations.replies` API で取得
5. メッセージテキストを結合
6. `RunCheckUseCase` を実行（source: 'slack', slackChannel, slackThreadTs を記録）
7. チェック完了後、`chat.postMessage` でスレッドに結果投稿:
   - 指摘件数サマリー（カテゴリ別）
   - 重要度別の件数
   - Web でチェック結果を確認するパーマリンク

### 非同期処理

- Vercel の `waitUntil` を使用
- Slack に即座に 200 応答を返しつつ、バックグラウンドでチェック実行
- チェック完了後に Slack にメッセージ投稿

## Stub 実装方針

全外部 API Gateway に Stub 実装を用意し、環境変数の有無で切り替える。

| Gateway | 本番実装 | Stub 実装 | 切替条件 |
|---------|---------|-----------|----------|
| AIGateway | AnthropicAIAdapter | StubAIAdapter | `ANTHROPIC_API_KEY` の有無 |
| EmbeddingGateway | OpenAIEmbeddingAdapter | StubEmbeddingAdapter | `OPENAI_API_KEY` の有無 |
| ArticleFetchGateway | NoteApiArticleFetchAdapter | StubArticleFetchAdapter | 常に本番（note API はキー不要） |
| Slack API | 直接利用（Composition 内） | ログ出力のみ | `SLACK_BOT_TOKEN` の有無 |

**Stub の返却値:**
- StubAIAdapter: カテゴリに応じた固定の指摘事項を返す
- StubEmbeddingAdapter: ランダムな 1536次元ベクトルを返す
- Composition 内の切替例: `const aiGateway = process.env.ANTHROPIC_API_KEY ? new AnthropicAIAdapter() : new StubAIAdapter()`

## 検証方法

1. **ローカル Supabase 起動**: `supabase start` → 接続情報を `.env.local` に設定
2. **DB セットアップ**: `prisma db push` → pgvector extension 有効化
3. **開発サーバー**: `pnpm dev`
4. **認証テスト**: `SKIP_AUTH=true` で開発用ログイン → 各ページ遷移確認
5. **チェック実行テスト**: 新規チェック → テキスト入力 → チェック実行 → 結果表示
6. **ルール管理テスト**: ルール追加 → チェックで反映確認
7. **ナレッジテスト**: 手動追加 → embedding 生成確認 → チェックで RAG 検索反映確認
8. **ソーステスト**: note ソース追加 → 記事同期 → 取込 → ナレッジ化確認
9. **品質チェック**: `pnpm verify`（lint → typecheck → build → unit test）
10. **E2E テスト**: `pnpm verify:full`（上記 + E2E テスト、実 AI Gateway + 実 DB）

### テスト環境と外部 API の関係

| テスト種別 | DB | 外部 API | 実行環境 |
|-----------|-----|---------|---------|
| Unit テスト | 不要 | Stub（モック） | CI / ローカル |
| Integration テスト | 実 DB（ローカル Supabase） | 実 API（キーがあれば） | ローカル / CI（secrets 設定時） |
| E2E テスト | 実 DB（ローカル Supabase） | 実 API（`ANTHROPIC_API_KEY` + `OPENAI_API_KEY` 必須） | ローカルのみ（CI では実行しない） |

E2E テストは `.env.local` に設定された実 API キーを使用し、実際の AI レスポンスで全パイプラインを検証する。
