# セキュリティ修正（CRITICAL / HIGH）

## 目的

記事の作例として公開するにあたり、読者からツッコミが入るレベルのセキュリティ問題を解消するため。

---

## 対応一覧

| # | 重要度 | 概要 | 対象ファイル |
|---|--------|------|-------------|
| 1 | CRITICAL | `SKIP_AUTH` に `NODE_ENV` ガードがない | middleware.ts, dev-login/route.ts, auth.composition.ts, layout.tsx |
| 2 | CRITICAL | API Route `/api/checks/[id]` に認証なし（IDOR） | api/checks/[id]/route.ts |

## 修正方針

### 1. `SKIP_AUTH` に `NODE_ENV` ガード追加

`process.env.SKIP_AUTH === 'true'` を判定しているすべての箇所に `process.env.NODE_ENV !== 'production'` を追加する。

**対象箇所（4 箇所）:**

- `src/middleware.ts:8`
- `src/app/api/auth/dev-login/route.ts:5`
- `src/backend/contexts/auth/presentation/composition/auth.composition.ts:26, 80`
- `src/app/(auth)/layout.tsx:15`

**方針:**
- ヘルパー関数 `isSkipAuth()` を `auth.composition.ts` に定義し、各箇所から参照する
- `isSkipAuth()` は `process.env.SKIP_AUTH === 'true' && process.env.NODE_ENV !== 'production'` を返す
- middleware は auth context の composition を import 可能（presentation 層の composition は外部からの参照を許可されている）

---

### 2. API Route `/api/checks/[id]` に認証追加

**現状:**
```
GET /api/checks/[id] → loadCheckDetail(id) → JSON 返却（認証なし）
```

**修正方針:**
- `getSession()` で認証チェック。未認証なら `401` を返す
- `loadCheckDetail` に `userId` を渡し、ユーザースコープで取得する（#4 と連動）

---

### 3. Server Action に認証チェック追加

**対象 4 ファイル:**
- `create-check.action.ts`
- `delete-check.action.ts`
- `retry-check.action.ts`
- `resolve-issue.action.ts`

**修正方針:**
- 各 action の先頭で `getSession()` を呼び出し、未認証なら `{ success: false, error: '認証が必要です' }` を返す
- `createCheck` では取得した `session.id` を `useCase.execute()` に `userId` として渡す
- `deleteCheck` / `retryCheck` / `resolveIssue` では、UseCase 内で対象リソースの `userId` が `session.id` と一致するか検証する（#4 と連動）

---

### 4. データアクセスにユーザースコープを追加

**変更対象:**

#### Repository 層

`PrismaCheckRepository` に以下の変更を加える:

- `findById(id, userId)` — `WHERE id = ? AND userId = ?` で取得
- `findAll(params)` に `userId` を必須パラメータとして追加 — `WHERE userId = ?` で絞り込み
- `delete(id, userId)` — `WHERE id = ? AND userId = ?` で削除（該当なしなら例外）
- `count(userId)` — `WHERE userId = ?` でカウント

#### Repository interface

`CheckRepository` interface の各メソッドシグネチャに `userId` パラメータを追加する。

#### UseCase 層

各 UseCase に `userId` パラメータを追加し、Repository 呼び出し時に渡す:

- `GetCheckDetailUseCase.execute(id, userId)`
- `ListChecksUseCase.execute(params)` の params に `userId` を追加
- `DeleteCheckUseCase.execute(id, userId)`
- `RetryCheckUseCase.execute({ checkId, sectionId, userId })`
- `ResolveIssueUseCase.execute({ checkId, issueId, userId })`
- `RunCheckUseCase.execute(params)` の params に `userId` を追加

#### Loader 層

- `loadCheckDetail(id, userId)` — userId を UseCase に渡す
- `loadCheckList(params)` — params に `userId` を追加し UseCase に渡す

#### Slack 経由のチェック

`HandleSlackMentionUseCase` から作成されるチェックは `userId: null` のまま（Slack ユーザーと webapp ユーザーの紐付けは現時点ではスコープ外）。Slack 経由のチェックは Web UI からはアクセスできなくなるが、これは想定通り。

---

### 5. dev セッション Cookie のバリデーション追加

**現状:**
`JSON.parse(raw) as DevSession` で型アサーションのみ。

**修正方針:**
- Zod スキーマ `DevSessionSchema` を定義し、`safeParse` でバリデーションする
- parse 失敗時は `null` を返す（既存の catch と同じ挙動）
- #1 の `NODE_ENV` ガードと合わせて、production では到達しない経路になる

---

### 6. `next.config.ts` にセキュリティヘッダー追加

以下のヘッダーを `headers()` で全ルートに設定する:

| ヘッダー | 値 |
|----------|-----|
| `X-Frame-Options` | `DENY` |
| `X-Content-Type-Options` | `nosniff` |
| `Referrer-Policy` | `strict-origin-when-cross-origin` |
| `Permissions-Policy` | `camera=(), microphone=(), geolocation=()` |

CSP と HSTS は Vercel のプラットフォーム側で設定するケースも多いため、ここでは追加しない。

---

### 7. `createCheck` の入力バリデーション追加

**現状:**
TypeScript の型アノテーションのみ。ランタイムバリデーションなし。

**修正方針:**
- Zod スキーマを action 内で定義し、`safeParse` でバリデーションする
- バリデーション対象:
  - `content`: 必須、最大 30,000 文字（domain model の制約と合わせる）
  - `platform`: オプション、enum (`youtube` | `x` | `note` | `other`)
  - `title`: オプション、最大 200 文字
- バリデーション失敗時は `{ success: false, error: '入力内容が不正です' }` を返す
